--- THE VOID OF INTENT ---
Intent has no mass.
Plans have no mass.
Claims have no mass.
Descriptions of what the agent "would do" have no mass.

Only these things count as observable matter:
- emitted final artifacts in AGENT OUTPUT,
- explicit intermediate state traces in AGENT OUTPUT,
- executable evidence in OBJECTIVE EXECUTION TRACE,
- file/artifact identifiers or values that appear in the task-visible record,
- multi-turn continuity that can be reconstructed from emitted state.

If a result is asserted but not grounded in observable matter, it is absent.
If a state transition is implied but not recorded, it is absent.
If the output says a tool was used but the execution trace does not show usable evidence, the tool use is absent.

--- STATEFUL BENCHMARK LAW ---
For multi-domain stem-agent episodes, success requires all of the following:
1. The output must address the exact task or episode contract.
2. The output must preserve state across the required sequence of turns or events.
3. The output must include enough intermediate trace to reconstruct how the final artifact was produced.
4. The output must ground claims in public artifacts or execution evidence.
5. The final answer must satisfy the deterministic verifier target implied by the task.

If any required state, action, artifact, or verifier evidence is missing, success is false.
There is no partial success.

--- DOMAIN-SPECIFIC OBSERVABLES ---
Trading floor episodes require:
- a transaction ledger with tick, asset, side, quantity, price, fee, and cash_after,
- final positions and final cash,
- no illegal trade relative to the visible rules,
- continuity from starting portfolio through every ledger row.

Security sandbox episodes require:
- inspected source or candidate evidence,
- a selected vector or packet,
- an observed sandbox result or verifier-compatible proof object,
- no claim that depends on external systems or non-local targets.

Matrix database episodes require:
- exact answer set,
- relation chain evidence,
- path trace for each returned node or record,
- filtering evidence for node properties and edge relations.

Legacy deterministic tasks require:
- exact final answer,
- executable or visibly reconstructable verification evidence,
- no unsupported numeric or factual assertion.

--- CRITIQUE RULES ---
The critique must be one sentence.
It must describe only the observed physical deficit.
It must not include advice, repair instructions, code hints, algorithm hints, or tool recommendations.

Good critique examples:
- "The output states a final portfolio but no ledger rows connect the starting portfolio to the claimed close state."
- "The output names a candidate packet but provides no sandbox observation or verifier-compatible proof object."
- "The output returns records without path traces through the required relation chain."

Bad critique examples:
- "The agent should parse the CSV with pandas."
- "Use a graph traversal algorithm next time."
- "It almost solved the task but forgot to run the verifier."
