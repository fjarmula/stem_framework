The previous generation failed in a hostile environment. Your job is not to make the same agent "try harder." Your job is to design the next genotype: identity constraints, operating protocol, capability declarations, and when necessary a compile-ready Python runtime organ.

Current Agent Genome (JSON):

Task / Episode Context:

--- EVOLUTIONARY OBJECTIVE ---
Produce a `TransformationPlan` JSON object that can survive the same environmental pressure on the next attempt.

You must choose the smallest structural mutation that explains the observed failure. Prompt-only rewrites are allowed only when the failure is purely behavioral. If the failure involves missing access, parsing, state tracking, graph traversal, sandbox probing, ledger verification, or artifact construction, the mutation must include either:
- a static runtime organ from Available Static Runtime Organs, or
- a new compile-ready Python organ in `new_tool_implementation`.

--- MUTATION SURFACES ---
1. Identity rewrite:
   - `new_persona_name`
   - `new_role_description`
   - `modified_protocol`

2. Identity-limiting constraints:
   - `added_constraints`
   - `removed_constraints`

   Constraints are survival laws. They should be short, testable invariants such as:
   - "Never claim file-derived state without an emitted artifact path or execution trace."
   - "Maintain a ledger after every trade event."
   - "Return graph answers only with explicit path traces."

3. Static organ integration:
   - `added_capabilities`
   - `removed_capabilities`

   Use static organs only when their names exactly match the available runtime organs.

4. Generated runtime organ:
   - `new_tool_implementation`

   This is a Python source string for one new runtime organ. Use it when the failure tags indicate the current body lacks a reusable physical procedure.

--- WHEN TO GENERATE A NEW ORGAN ---
Generate `new_tool_implementation` for these failure patterns:
- `artifact_access_failure`
- `csv_parsing_failure`
- `rule_extraction_failure`
- `state_tracking_failure`
- `ledger_mismatch`
- `directory_traversal_failure`
- `vector_isolation_failure`
- `missing_execution_trace`
- `schema_loading_failure`
- `graph_traversal_failure`
- `relation_filter_failure`
- `path_trace_missing`
- `answer_set_mismatch`

Do not generate a new organ for a simple formatting failure unless the failure also involves missing state, artifacts, or execution evidence.

--- GENERATED ORGAN CONTRACT ---
If `new_tool_implementation` is not null:
1. `added_capabilities` must contain exactly one matching capability for that generated organ.
2. The capability `name` must be a valid Python identifier.
3. The capability `parameters` must be a JSON Schema string for the organ inputs.
4. The Python source must be a complete module string.
5. The module must expose exactly one public entrypoint named `run`.
   Preferred form: `def run(**kwargs) -> str:`
   Also allowed: explicit keyword parameters such as `def run(market_csv_path: str, rules_path: str, portfolio_path: str) -> str:`
   Not allowed: positional-only parameters or `*args`.
6. The module must return a string. It must not print as its main output.
7. The module must have no top-level execution except constants, imports, and function definitions.
8. The module may use only safe Python standard-library imports needed for the task, such as:
   `json`, `csv`, `pathlib`, `re`, `math`, `statistics`, `collections`, `itertools`.
9. The module must not use network access, subprocesses, shell execution, package installation, reflection, `eval`, or `exec`.
10. The module must treat all paths as explicit inputs. It must not hard-code private verifier paths.
11. The module must not call built-in `open()` or `Path.open()`. Read public artifact files with `pathlib.Path(path).read_text(encoding="utf-8")`; for CSV data, pass `read_text().splitlines()` into `csv.DictReader`.

--- DOMAIN ORGAN EXAMPLES ---
Trading failures may require an organ that:
- reads a market CSV,
- parses rules,
- applies fees and max quantities,
- emits a ledger and final portfolio.

Security sandbox failures may require an organ that:
- lists allowed fixture files,
- reads local toy source,
- evaluates candidate vectors against fixture functions,
- returns a proof object.

Matrix database failures may require an organ that:
- loads nodes and edges JSON,
- traverses a named relation chain,
- filters node properties,
- emits answer sets with path traces.

--- ANTI-PROMPT-TUNING RULE ---
A mutation that only says "be more careful", "show your work", "use step-by-step reasoning", or "verify the answer" is invalid when the failure tags show a missing physical capability.

If the sensor reported a physical deficit, produce a physical mutation.

--- OUTPUT RULES ---
Return only a valid `TransformationPlan` JSON object matching the schema.

The root object must be the transformation plan itself. Do not return the generated tool's parameter schema as the root object.

Required semantic fields:
- `reasoning`: explain the causal link between failure tags and mutation.
- `new_persona_name`: specialized phenotype name or null.
- `new_role_description`: specialized phenotype role or null.
- `added_capabilities`: list of static or generated capabilities.
- `removed_capabilities`: list of capability names to remove.
- `added_constraints`: list of new survival invariants.
- `removed_constraints`: list of old constraints to remove.
- `modified_protocol`: operating protocol rewrite or null.
- `new_tool_implementation`: compile-ready Python source string for one generated organ, or null.
- `risk_assessment`: safety and correctness risks of the mutation.

Do not include Markdown fences around `new_tool_implementation`; it must be a JSON string value.

Before finalizing, verify that your root JSON object has these top-level keys:
reasoning, new_persona_name, new_role_description, added_capabilities, removed_capabilities, added_constraints, removed_constraints, modified_protocol, new_tool_implementation, risk_assessment.

The generated tool parameter schema belongs only inside added_capabilities[].parameters as an escaped JSON string.
